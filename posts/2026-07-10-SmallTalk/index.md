개발을 할때 보통 설계를 머리든, 실제 기록이든 하고 진행하게 된다. 이때 개발을 하다가 다음과 같은 양자 택일에 대해서 생각했다.

- 설계를 할때부터 최대한 생각하는 시간을 가져서 최대한 최적화된 코드를 한번에 내는것
- 우선 작동하는 매커니즘으로 작동을 확인한 뒤, 최적화를 진행하여 완성본을 내는 방식

이때 서로 장단점이 있다고 생각한다.

## 설계를 완성도를 우선시 할때
**장점**
1. 재작업 최소화
이미 코드 단계에서 필요 없는 코드를 덜어내었기 때문에, 이후 작업에서 해당 부분이 병목을 일으킬 가능성이 낮아진다.
2. 코드 일관성
작업자가 짜는 코드 스타일이 반영되서 작성될 것이기 때문에 타 스크립트들과 비슷한 형태가 될 것이고, 이는 코드 리뷰시 좀더 원활하게 할 수 있도록 도움을 준다.

**단점**
1. 오버엔지니어링
최적화를 하고 보면 사실상 의미가 없는 최적화, 혹은 사용하지 않는데 개발 자원을 사용한 결과가 될 수 있다.
2. 환경 변화에 취약
당시 상태에서는 최적의 형태였을지 모르지만, 이후 다른 코드들이 작성되고 난 이후 확인 하였을때 최적의 형태가 아닐 수 있다.
3. 순수 시간 자원 낭비
업무시에 필요한 기능을 만드는데, 시간을 오래 소비하게 되며, 이는 타 작업의 지연을 가져오는 결과가 될 수 있다.

## 구현 속도를 우선시 한 뒤에 최적화 작업을 할시
**장점**
1. 불필요한 개발 자원 낭비 방지
당장 머리속에서 작동하는 로직을 구현하기 때문에, 막히는 부분이 거의 없이 진행할 수 있음
2. 측정 기반 최적화
프로파일링을 해서 실제로 필요한 부분을 찾아내서 최적화 하는 방식을 적용할 수 있다.
3. 착수 난이도가 낮아짐
일단 당장 작업을 이어나갈 수 있기 때문에 막히는 부분 없이 작업을 이어나갈 수 있다.

**단점**
1. 재작업 리스크
초기 구조가 무너졌을시 최적화가 아니라 다시 구현해야 할 상황도 생길 수 있다.
2. 방치시 리스크
어쨋든 작동한다 라는 부분이기 때문에, 이후 방치되어 병목 지점으로 남아버릴 가능성이 있다
3. 일관성 저하
코드를 작성시 임기 응변적 코드 문법이 생겨 여러곳의 스크립트들이 서로 다른 형태와 패턴을 사용하여 가독성이 떨어질 수 있다.


### 결론은 무엇인가
사실 코드를 빠르고 정교하게 적는게 최선책이라 생각하지만, 실제 그렇게 하는 것은 매우 어렵고, 많은 기반 지식을 요구하는 것이다. 그렇기에 프로그래머들은 언제나 저울질을 하면서 코드를 작성한다 생각한다.

물론 두 방법 모두 흑백 논리로 좋다 나쁘다를 말하는 것은 불가능 하다고 생각한다. 왜냐하면, 어떤 경우에는 작동만 해도 괜찮은 상황이 있을것이고, 어느 경우에는 최대한 정교하게 설계해야 할때가 있을 것이기 때문이다.

그렇지만, 일반론적으로, 그리고 실제 업무시에 경우에는 후자쪽이 좀 더 알맞은 방식이지 않을까 라는 생각을 하게 되는 것 같다.
